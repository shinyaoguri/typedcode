# ADR-0008: 試験モードはフルスクリーンを要求するが強制せず、状態を記録する

- **Status**: Proposed
- **Date**: 2026-06-05
- **Deciders**: (PR 上の合意者 / レビュアー)
- **PR / Commit**: (this PR)

> ADR-0007 は捕捉 (passive) に限定し「**アクティブ強制は対象外**」と明記した。本 ADR はその穴を埋める = exam モードがウィンドウ環境に**能動的に働きかける**方針を確定する。結論を先に: **フルスクリーンを要求するが強制しない (非フルスクリーンでも全機能利用可・常時警告)。フルスクリーン状態はハッシュチェーンに記録する。**

## Context

ADR-0006 で exam モードの問題束縛、ADR-0007 で生信号捕捉を確定した。残るのが「ウィンドウ環境に対してツール側がどこまで能動的に介入するか」。前提となる制約:

- **純Web の限界 (ADR-0007 と同じ誠実さで明記)**: ブラウザのページは OS ウィンドウマネージャを支配できない。
  - 「常に最前面に固定」= **API が存在しない (不可能)**。
  - 「強制的に最大化」= 検出も強制も確実な API なし。
  - 「Alt+Tab / Cmd+Tab をブロック」= OS レベルなので不可。
  - → **唯一の実用レバー = Fullscreen API**。
- **Fullscreen すら「強制」はできない**: 開始にユーザ操作 (gesture) が要り、**Esc でいつでも抜けられる (ブラウザが保証する逃げ道、奪えない)**。
- **別端末問題は対象外**: 手元のスマホ等で生成 AI を引く経路はブラウザでは原理的に防げない。→ **運用 (試験監督) でカバーする前提**とし、本 ADR では扱わない。
- **「止めない」原則 (上位制約)**: ネット不安定・100 台同時でも正規受験者を絶対に締め出さない。サーバ非依存・ローカル完結。
- **"紙のように" 思想**: 防御の本体は監督 (唯一の本物の air gap) ＋ 証拠。ツールの役割は**安易な不正の摩擦化と、消えない証拠化**であって、不正の物理的封鎖ではない。

## Considered Options

### Option A: ハードロックダウン (kiosk / LockDown Browser / MDM 管理端末)
- Pros: 実際に alt-tab やタブ移動を封鎖できる。
- Cons: 「**ただブラウザで使うだけ**」「**端末を個別認識しない**」という本プロジェクトの根本制約を破る。配布・運用コストも別物。→ **却下**。

### Option B: フルスクリーン必須 (非フルスクリーンでは編集ロック / 開始不可)
- Pros: 抑止力は最も強い。
- Cons: (a) **Esc で必ず抜けられる**ため「必須」を技術的に保証できず、freeze ↔ 再要求の応酬になる。(b) フルスクリーン非対応・環境差・誤操作で**正規受験者を締め出すリスク**。(c) 100 台同時で 1 人でも詰むと監督対応コストが跳ねる。→ **「止めない」原則と相性が悪く却下**。

### Option C: フルスクリーンを要求するが強制しない (警告のみ) ＋ 状態を記録
- Pros: ブラウザの限界 (強制不可) と「止めない」制約に正直。**正規受験者を絶対に締め出さない**。安易な不正 (同一 PC の alt-tab) には警告という摩擦と、チェーンに刻まれる消えない証拠で十分に対処。判定は後段 (監督＋分析) に委ねられる。
- Cons: 抑止力は限定的 (警告は無視できる)。→ **これは設計上の割り切り**。"紙のように" の本体は監督＋証拠なので許容。**採用**。

## Decision

exam モードに以下を確定する。casual モードは従来どおり何もしない (モード差は機能のみ、の原則を維持)。

1. **要求**: 開始儀式で `requestFullscreen()` を要求する (開始ボタンの user gesture に紐付け)。
2. **強制しない**: 非フルスクリーンでも**全機能を利用可**。ただし非フルスクリーン中は**非ブロッキングの常時警告表示** (バナー / インジケータ) を出し、ワンクリックでフルスクリーン復帰 (再 gesture) できるボタンを添える。**編集ロックや開始不可にはしない**。
3. **記録**: フルスクリーン状態を ADR-0007 の捕捉規約 (ハッシュチェーン格納・graceful absence・加算的 versioned スキーマ) に準拠して記録する。
   - 開始時の **grant / deny** (`requestFullscreen()` の解決 / 失敗)。
   - **`fullscreenchange` ごとの enter / exit 遷移** (タイムスタンプ付き)。
   - **非フルスクリーン滞在区間** (開始・復帰・継続時間) → 「フルスクリーン外に何秒いたか」を後段で集計可能にする。
   - Fullscreen API 自体が無い環境は値を捏造せず **`unavailable` を事実として記録** (可用性も信号)。
4. **Keyboard Lock / Pointer Lock は使わない**: 強制しない方針では Esc を奪う意味がない。将来 strict ポリシーを足すなら別判断 (本 ADR は warn-only を確定する)。

記録は ADR-0007 Tier 0 C 群 (liveness / ギャップ) に「fullscreen 状態」を加える形。具体的な event-type / field は `system-spec.md` で確定 (ADR-0007 のスキーマ作業と同時)。

## Consequences

### Positive

- ブラウザの限界と「止めない」制約に正直で、**正規受験者を絶対に締め出さない**。
- 安易な同一 PC 不正に**摩擦 (警告) ＋ 消えない証拠 (チェーン記録)** を付与できる。
- 試験監督が「警告が出っぱなしの端末」を**目視で発見**する補助になる。
- ADR-0007 の捕捉機構をそのまま再利用 (新しい格納機構は不要)。

### Negative / Trade-offs

- **抑止力は限定的**: 警告は無視できる。これは「強制ではなく証拠」という設計上の割り切り。
- **別端末は防げない** → 運用 (監督) 依存を明示的に受け入れる。
- **警告 UX 設計が必要**: 鬱陶しすぎず、かつ無視されすぎない塩梅。
- フルスクリーン非対応 / 拒否環境では警告が出っぱなしになる → 運用要項で説明する。

### Follow-ups / 残課題

- **警告 UI の具体設計** (文言・i18n `ja`/`en`・常時バナー or トースト・復帰ボタン)。
- **fullscreen 状態の event-type / field を `system-spec.md` に定義** (ADR-0007 のスキーマ作業と同時)。
- 「非フルスクリーン累計時間」を proof サマリ / 検証 UI に出すか (後段の表示・分析の話)。
- 将来 **strict モード** (離脱即失格 / freeze) を出すなら別 ADR。本 ADR は **warn-only を確定**する。
