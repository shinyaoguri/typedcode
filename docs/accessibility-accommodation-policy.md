# 支援技術・IME 配慮ポリシー (★6b)

TypedCode の捕捉・分析・コホート基準が、**IME (各言語の入力方式) や支援技術 (assistive technology) の利用者を不当に不利にしない**ための運用方針。これは UX の細部ではなく**公平性・アクセシビリティの一次要件**であり、誤フラグは差別になりうる。

> **前提となる決定**: 分析・コホートは常に advisory で**判定しない** (ADR-0009/0023)。外れ値は triage であって違反ではない (ADR-0025)。本ポリシーはそれを支援技術・IME の文脈で具体化し、**捕捉層の実挙動を正直に開示**して採点者が正しく解釈できるようにする。

---

## 0. 原則 (決定)

1. **支援技術・IME の利用を不利にしない。** 分析・コホート基準は、これらの利用者を「異常」「自動化」「非ピュアタイピング」として**自動的に不利に扱ってはならない**。最終判断は常に人間 (採点者) にあり、配慮事項を踏まえて行う。
2. **障害情報を収集しない・AT を自動検出して疑わない。** TypedCode は障害の有無や支援技術の種類を**データとして集めない**。配慮は教員・教育機関が**TypedCode の外で**手配する (標準的なアクセシビリティ実務)。ツールは中立に留まる。
3. **捕捉層の実挙動を正直に開示する。** 後述の「既知の偽陽性モード」を採点者・運用者に明示し、`isPureTyping=false` や分析 signal を**機械的にクロと読ませない**。
4. **記述的であって規範ではない** (ADR-0025 再掲)。IME・支援技術・タイピング速度の多様性は正当な分布の幅であり、コホート基準の外れ値はそれ自体では何も意味しない。

---

## 1. 捕捉層の実挙動 (正直な開示)

`isPureTyping` は `InputTypeValidator` の許可/禁止リストが唯一の真実 (ADR-0005)。アクセシビリティ観点での実態:

### IME (日本語/中国語/韓国語などの変換入力) — 不利にならない
- `insertCompositionText` / `deleteCompositionText` / `insertFromComposition` は**許可リスト**にある。→ **IME は `isPureTyping` を false にしない。** 捕捉層では通常タイピングと同等。
- リスクは**分析層のみ**: 変換確定でまとまった文字が一度に入る・1 文字あたりの離散打鍵が少ない・打鍵動態が Latin と異なる。これらが focus-burst (バースト挿入) や将来の打鍵動態系アナライザを誤って踏みうる。→ §2 の偽陽性モードで扱う。

### ディクテーション / 音声入力 / 自動修正 / 単語予測 — 不利になりうる (要注意)
- これらは環境により `insertReplacementText` で挿入することがある。`insertReplacementText` は**禁止リスト**にある。→ **`isPureTyping` が false になりうる。**
- つまり **`isPureTyping=false` は「ペースト等の不正」とは限らず、「支援技術による入力」でもありうる。** 両者を捕捉層は区別できない (ブラウザが提供する inputType が同じ)。これは検出で塞げる穴ではなく**原理的な曖昧性**。
- 帰結: `isPureTyping=false` / 「外部入力」signal を**単独でクロと解釈してはならない**。過程リプレイ (ProcessSummary の見どころ) と本人の振り返り (reflectionNote)、そして事前の配慮申請を併せて人間が判断する。

### スイッチ/スキャン入力・視線入力・オンスクリーンキーボード — 分析層で要注意
- 入力が非常に遅い・長い停止が入る → `pauseCount` / `longestPauseMs` が大きくなる。まとまって入力されると focus-burst を踏みうる。いずれも **advisory** で valid 不反映。

## 2. 既知の偽陽性モード (採点者向け早見表)

| 配慮 | 捕捉/分析での見え方 | 正しい解釈 |
|---|---|---|
| IME 変換 | バースト挿入 (focus-burst)・離散打鍵が少ない | 正当な入力方式。`isPureTyping` は true のまま |
| ディクテーション/音声入力 | `isPureTyping=false` (`insertReplacementText`)・「外部入力」signal | 不正とは限らない。配慮申請+過程で判断 |
| 自動修正/単語予測 | `isPureTyping=false`・まとまった置換 | 同上 |
| スイッチ/視線/低速入力 | 長い停止 (`pauseCount`)・バースト | 正当。速度の多様性であって異常ではない |
| スクリーンリーダー併用 | focus/blur の増加 (`focusLossCount`) | 支援ソフト操作由来。離脱=不正ではない |

**コホート基準 (ADR-0025) との関係**: 上記はいずれもコホート分布の正当な外れ値になりうる。AT/IME 利用者を「外れているから精査対象」と扱うのは差別。基準は記述的に用い、配慮事項と併読する。

---

## 3. 試験モードとの緊張 (運用指針)

- 試験モード (ADR-0006/0008) は最も厳格で、ペースト系入力を warning/制約とする。ディクテーション・単語予測が `insertReplacementText` を使うと、これらに依存する受験者が**構造的に不利**になりうる。
- **解決はツールではなく運用**: 教員・教育機関が事前に配慮を手配する — 代替評価、AT の事前承認、または試験モードを使わない選択。TypedCode は配慮の可否を**裁定しない**。
- 事前申請があった受験者については、採点者が `isPureTyping=false` や外部入力 signal を**その文脈で**解釈する (機械的減点をしない)。

---

## 4. やらないこと (非目標)

- 障害・支援技術の有無を**データ収集しない** (proof にもメタにも焼かない)。
- 支援技術を**自動検出して「疑わしい」に分類しない**。
- `insertReplacementText` 等を「支援技術だから許可」と緩めることも**しない** — ペースト回避の抜け穴になるため。曖昧性は**検出ではなく人間の解釈**で吸収する。

---

## 5. 将来課題

- 捕捉層がプラットフォームから支援入力と敵対的ペーストを区別できる信号を得られるなら、偽陽性の一部を減らせる可能性 (ただし現状ブラウザの inputType では区別不能 — 期待しない)。
- ★5 コホート基準の実装時、AT/IME 利用者を基準母集団でどう扱うか (除外/別基準) の具体化。
- 配慮申請の受け渡し (ツール外) を採点ワークフローにどう接続するかの運用テンプレート。
